# Phase 0 Observations

## 我們做出了什麼？

- 首頁已顯示 Phase 0 原始資訊列表，並把每筆資料的來源、查核狀態、更新時間與「不準確或待確認訊號」放在同一張卡片上。
- 整理工作台仍保留保守安全預設：候選類型待判斷、信心程度低、不可直接行動。
- 目前的整理只根據原文詞句與 `verificationStatus` 標示問題，不代表完成事實查核。

## 哪些資料讓 agent 很容易整理？

- M-009 和 M-010 有較明確的時間、地點或現場更新語氣，容易讓 agent 想整理成候選結果。
- 但這兩筆仍是 `needs_review`，不能因為文字看起來完整就顯示為已確認。

## 哪些資料讓 agent 亂補或誤判？

- M-001 的「老雜貨店後面」可能讓 agent 自動補成具體地址，但原文沒有地址。
- M-005 的來源型別是 `official_notice`，但原文說不知道截圖是哪一天、也不知道是不是官方公告，不能直接當官方公告。
- M-012 看起來像求助，但來電者不在現場，也無法確認親友目前位置，不能自動建立任務。

## 哪些資訊不能直接變成任務？

- M-001 缺精準地點與需求確認，不能直接派十幾個人。
- M-006 對集合點是否安全有衝突說法，不能直接請人前往停留。
- M-008 只有「先不要再派人」，但原因不明，不能轉成可執行任務。
- M-011 涉及長者住家位置與公開同意，不能補出完整地址或直接公開派工。
- M-012 位置、當事人狀態與是否要建立任務都未確認。

## 哪些資訊的操作者不是當事人？

- M-001、M-004、M-007 來自社群或群組轉述，操作者不一定是需求當事人。
- M-011 是現場志工代長者轉述，還需要確認長者同意與需求細節。
- M-012 是外地家屬來電，不在現場且無法確認親友目前位置。

## 我們剛剛偷偷假設了什麼？

- 假設「看起來像現場更新」就比較可靠，但仍需要查核狀態與來源交叉確認。
- 假設相對地點足以派工，但「後方」「附近」「第二排住家」都不是可安全行動的地點。
- 假設資源或人力狀態還有效，但原文多處提到昨天、下午、中午前、原本單據可能沒更新。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要人工確認角色，負責補問來源、時間、地點、當事人同意與目前狀態。
- 需要更清楚的整理草稿欄位，把「原文可見」和「agent 推測」分開。
- 需要讓使用者能標記「暫時不要使用」「補問資訊」「建立候選通報」等下一步，但不能自動派工。

## 我們需要更清楚資料格式的地方

- 來源型別不等於查核狀態；例如 `official_notice` 仍可能是未確認截圖。
- 需要分開記錄事件發生時間、回報時間、最後確認時間。
- 需要欄位記錄地點精準度、是否有當事人同意、是否為轉述。

## 我們需要人工確認的地方

- 地點：相對位置、集合點、安全性、是否適合停留。
- 時間：資源數量、人力狀態、道路或公告是否仍有效。
- 當事人：需求是否仍存在、是否同意公開住址或建立任務。
- 衝突：不同來源互相矛盾時，誰有權更新或撤回舊資訊。
